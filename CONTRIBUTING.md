# Contributing to @heyai-rules/heyai-ruleset 🚀

First off, thank you for considering contributing to `heyai-ruleset`! It's people like you that make the open source community such an amazing place to learn, inspire, and create.

## 🌈 Code of Conduct
This project and everyone participating in it is governed by the [Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.

## 🛠️ How Can I Contribute?

### Reporting Bugs
- Check the [Issues](https://github.com/Arter2507/create-heyai-ruleset/issues) to see if the bug has already been reported.
- If not, create a new issue using the **Bug report** template.

### Suggesting Enhancements
- Check the [Issues](https://github.com/Arter2507/create-heyai-ruleset/issues) to see if someone else has already suggested it.
- If not, create a new issue using the **Feature request** template.

### Pull Requests
1. **Fork** the repository to your own account.
2. **Clone** your fork to your local machine.
3. **Create a new branch** for your work (e.g., `feat/new-rule-kit` or `fix/cli-error`).
4. **Implement your changes** and add tests if applicable.
5. **Ensure all tests pass** (`npm test`).
6. **Commit your changes** using [Conventional Commits](https://www.conventionalcommits.org/).
7. **Push** your branch to your fork.
8. **Submit a Pull Request** against the `main` branch of the original repository.

## 🤝 Standards
- **Conventional Commits**: Every commit must follow wait `feat:`, `fix:`, `docs:`, etc.
- **Bilingual Support**: All documentation must be provided in both **Vietnamese** and **English**.
- **Security**: Never commit secrets or sensitive data. All contributions are scanned for malware and backdoors.

## 📝 PR Review Process
- All PRs require at least one approval from a maintainer.
- Continuous Integration (CI) checks must pass.
- We may ask for changes or clarifications before merging.

Thank you for your help!
